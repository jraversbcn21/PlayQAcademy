# AGENTS.md — PlayQAcademy / main

## Project overview

PlayQAcademy is a bilingual (ES/EN) learning platform for software QA professionals, built with **Next.js 15 (App Router), TypeScript (strict), Firebase (Auth + Firestore), and Tailwind CSS**. Content is organized into three active, self-contained campuses:

| Campus (`id`) | Modules | Lessons | Exams | Badges |
|---|---|---|---|---|
| QA Fundamentals (`qaFundamentals`) | 10 | 45 | 11 | 11 |
| ISTQB CTFL Foundation (`istqb`) | 6 | 25 | 9 | 15 |
| Playwright Automation (`automation`) | 8 | 41 | 4 | 5 |

Each module holds bilingual lessons (flashcards, quizzes, exercises, tables, callouts…). Campuses have associated exams. Gamification (points, levels, badges, streaks) tracks user progress across all campuses. The global badge catalog is **48** (campus badges above + cross-campus badges: first-login/lesson/quiz/exercise, streaks, perfectionist, etc.).

## Conventions & workflow

- **Bilingual everywhere:** every user-facing string is `{ es, en }` (the `Bilingual` type). No empty `"en": ""`, no hydration-flash placeholders.
- **Direct-to-`main` workflow:** work is committed straight to `main` (no open PRs/branches).
- **Quality gate:** `npm run typecheck`, `npm run lint` (both **0 errors/warnings**), **and `npm run build`** before every commit. Build is non-optional, not just a pre-deploy spot-check — typecheck/lint alone have already missed a real `next build` failure once (the `useSearchParams()`/`Suspense` requirement, see Architecture note below).
- **Sync this file:** every feature commit is followed by (or includes) a `docs:` commit that updates AGENTS.md and references the feature hash. AGENTS.md describes the *current* state; git holds the changelog.

## Architecture (durable facts — read before changing content/exams/badges)

- **Campuses are a derived grouping over modules — NOT nested in data.** Progress/exams/gamification/badges key off `moduleId`/`examId`, never `campusId`. Adding or moving content needs no Firestore migration and breaks no URLs. Root `/[lng]` is the neutral "QA Campus" hub; sub-campus URLs are flat `/campus/[campusId]`. The single fixed parent abstraction lives in `src/types/campus.ts` + `src/lib/constants/campuses.ts`. (Full restructure history: `docs/todo-qa-campus-restructure.md`.)
- **Exams are data-driven.** `getExamsForCampus(campusId)` returns exams whose `moduleIds` ⊆ the campus modules. `isExamReady(exam)` gates "Coming Soon" and requires **both** enough total questions (`getAvailableQuestionCount(exam) >= exam.questionCount`) **and** every one of `exam.moduleIds` contributing at least one question — without the second check, a multi-module exam (midterm/final, mock exams) could hit its total purely from a few well-stocked modules while a required module sits at zero, silently never drawing from it despite the exam claiming full coverage. An exam with no/short/partial bank surfaces but is not startable, and lights up automatically once every required module has a bank (no code change). `generateExamQuestions` produces a **seeded (userId+examId), stable** selection at a **40/35/25** easy/medium/hard split. Each attempt persists its exact `questionIds` (display order) on the `exam_attempts` doc at `startExamAttempt` time; `take`/`submit`/`results` all resolve the question set via `resolveAttemptQuestions` (`useExamAttempt.ts`), which uses the persisted ids when present and only falls back to re-running the seeded `generateExamQuestions` selection for attempts that predate this field (or whose ids no longer fully resolve in the bank) — otherwise a bank growing after an attempt would silently change what that attempt's review shows. Fixed 2026-07-11 (audit finding, Fase 4; commits `00595e6`, `21663f3`); live-verified in production, see "Verified in browser" #19.
- **Question-bank registry.** Each `src/lib/constants/examQuestions/*.ts` calls `registerQuestions()` into a global bank and is side-effect-imported in `exams.ts`. `getQuestionsForModules` matches a question if **any** of its `moduleIds` is requested (`.some()`). To keep per-module exams **pure**, tag each question to a **single** chapter/module — broad multi-module tagging pollutes per-module pools.
- **Scoring is equal-weight:** `calculateScore = correct / total questions × 100`. The results page derives `correct` and `total` from the same question set, so the fraction and headline % always agree. (Old attempts stored before this keep their legacy score in Firestore; not recomputed.)
- **Badges are criteria-based** (`src/lib/constants/badges.ts`, `BadgeCriteria` union in `gamification.ts`). `module_completed` and `exam_passed` award through the data-driven flow (`badgeChecker.ts` + `recordExamPassed` / module-completion context) — **adding a badge object is enough, no new wiring**. Passing an exam queues a `BadgeUnlockedModal`. Only passes/completions *after* a badge is added award it (no retroactive award) — `badgeChecker.ts` only evaluates badges not already in `earnedBadges`, so changing a badge's criteria never revokes one a user already has. **Badge counts are always derived from real content/state, never hardcoded** — `perfectionist`'s criteria count reads `TOTAL_QUIZ_SECTIONS` (computed in `src/lib/constants/lessons/registry.ts` by counting `type === "quiz"` sections across every lesson) instead of a frozen number that silently went stale (44 vs. the real 82) as content grew. `speed_learner` ("Aprendiz Veloz") uses `{ type: "lessons_in_one_day", count: 5 }` — 5 lessons completed in one local-timezone day, tracked via `lessonsToday`/`lessonsTodayDate` on the gamification doc (`markLessonComplete` in `useLesson.ts`) — replacing an earlier `moduleCompletionTimes`-based criteria that no caller ever populated, making the badge (and `all_badges`, which required it) permanently unearnable. `perfect_module` is now "Perfeccionista Avanzado" (`perfect_quizzes` count 15) — previously identical to `perfect_5`'s criteria (both count 5), so they always awarded together despite different descriptions. `checkAndAwardBadges` falls back to `setDoc(..., { merge: true })` if `updateDoc` fails (doc doesn't exist yet) instead of swallowing the error — a badge is never reported to the caller (which triggers `BadgeUnlockedModal`) unless the write actually succeeds.
- **`campus_completed` criteria is scoped, never global.** It takes a `campusId` and checks only that campus's `getModulesForCampus(campusId)` — there is deliberately no "all modules across every campus" variant; a global version (`all_modules_completed`, no campusId) existed before the multi-campus expansion and silently became "complete all 24 modules in all 3 campuses" once QAF/ISTQB were added, which is why `all_modules_complete` ("Graduado PlayQAcademy") and `playq_certified` now use `campus_completed`/`exam_passed` instead. Don't reintroduce an unscoped variant without scoping it to a campus.
- **`exercise_completed` is a platform-wide counter, not campus-scoped** (no per-campus tracking of which exercise IDs belong to which module exists). It's correct today *only* because the `"exercise"` lesson-section type is used exclusively by the 8 Automation modules (one each) — `automation_master`'s `count: 8` happens to equal both "all exercises in Automation" and "all exercises on the platform". If QA Fundamentals or ISTQB ever add an `"exercise"` section, this badge's criteria silently stops meaning "Automation Master" and must be re-scoped first (see the comment on this case in `badgeChecker.ts`).
- **`all_badges_earned` criteria** (used by `all_badges`, "Coleccionista de Insignias"/"Badge Collector") checks that every *other* badge (`BADGES.filter(b => b.criteria.type !== "all_badges_earned")`) is in the user's `earnedBadges` — it excludes itself structurally rather than by hardcoded id, so it stays correct even if a second such badge were ever added.
- **Glossary** (`src/lib/constants/glossary.ts`): tagged by `chapter` — ISTQB uses bare keys `"1"…"6"`, QA Fundamentals uses `qaf-1…qaf-10`, Automation uses `auto-1…auto-8`. `GLOSSARY_CHAPTERS_BY_CAMPUS` is the explicit, correctly-ordered (not `.sort()`-derived) registry mapping each campus to its chapter keys in real module order — the `/glossary` page reads this directly instead of deriving order from the data, avoiding a lexicographic-sort bug (`"qaf-10"` sorting before `"qaf-2"` as strings). Section titles live in `CHAPTER_TITLES` (no campus-name prefix — the page's accordion section already provides that context). The page itself is a single-open campus accordion (same pattern as `/exams`): opening a campus shows its chapters as headed sections; typing in the search box bypasses the accordion entirely and shows matches from all 3 campuses grouped by campus header. Reachable from the navbar (`nav.glossary`, between Curriculum and Playground, desktop + mobile).
- **Module locking is OFF** (`ENFORCE_MODULE_LOCKING = false`): no module is ever `"locked"`; unlock logic never crosses campus boundaries.
- **Module `estimatedMinutes` is a manual value** shown directly in the UI (campus cards, dashboard, learn page) — bump it when you add/remove lessons in a module.
- **Legal pages** (`/privacy`, `/terms`, `/cookies`; content lives in `public/locales/{es,en}/common.json` under `privacy`/`terms`/`cookies` keys, rendered by `src/app/[lng]/{privacy,terms,cookies}/page.tsx`): `terms.s2Body` describes all three campuses (QA Fundamentals, ISTQB CTFL prep, Playwright automation) — update it if a campus is added/removed/renamed. `terms.s8Body` explicitly disclaims any ISTQB® affiliation/accreditation (PlayQAcademy's "Simulacro ISTQB CTFL" exams are unofficial practice material, not the real certification) — keep this if more third-party-branded mock content is ever added. Bump the relevant page's `lastUpdated` only when its body content actually changes.
- **Every auth-gated client page needs both halves of the guard.** The `if (authLoading || !user) return <spinner>` pattern (dashboard, exams, learn, badges, settings, leaderboard) only resolves once `authLoading` flips to `false`; if there's no signed-in user it stays `true` forever unless the page also has a `useEffect` that `router.push`es to `/auth/sign-in` once `!authLoading && !user`. `middleware.ts`'s `PROTECTED_PATTERNS` (cookie-based, server-side) is the other half — a path missing from both means the page spins indefinitely instead of bouncing to sign-in. `exams/[examId]/start` and `exams/[examId]/take/[attemptId]` gated the redirect on bare `!user` (missing the `authLoading` half) until fixed 2026-06-20 — see "Verified in browser" #10 for the hard-navigation bug this caused and the fix. `/badges` had the client-side half but was missing from `PROTECTED_PATTERNS` (the server half) until fixed 2026-07-10 — an audit finding, not user-reported; check both halves whenever a new gated page is added.
- **Lesson prev/next navigation is campus-scoped, like breadcrumbs and unlock logic.** `useLessonNavigation` (`useLesson.ts`) derives the previous/next module boundary from `getCampusForModule(moduleId)` + that campus's `getModulesForCampus(campusId)` order, returning `null` at a campus's first/last lesson — it used to walk the global `CURRICULUM` array's `order ± 1`, so the last lesson of Automation linked straight into the first lesson of ISTQB. Fixed 2026-07-10 (audit finding).
- **Streak day-boundary uses the LOCAL date, never `toISOString()`.** `src/lib/utils/date.ts` exports `localDateStr()`/`localYesterdayStr()` (built from `getFullYear`/`getMonth`/`getDate`, not UTC) — `useGamification.ts`'s `todayStr`/`yesterdayStr` and `useLesson.ts`'s lesson-completion timestamp delegate to them. `toISOString().slice(0, 10)` flips to "tomorrow" at 00:00 UTC (mid-evening in Spain), which broke streaks unfairly. Fixed 2026-07-10 (audit finding); any new day-boundary logic must use this helper, not `toISOString()`.
- **Firestore rules must be *deployed*, not just committed.** `firestore.rules`' `exam_attempts` update rule allows three status transitions: `in_progress→in_progress` (per-question answer saves), `in_progress→submitted`, `in_progress→abandoned`. Editing `firestore.rules` and committing it does **nothing live** until `npx firebase deploy --only firestore:rules --project playqa-academy` actually runs — a 2026-06-20 session found the `in_progress→in_progress` transition had been in source since commit `5f66750` but was never deployed, so every per-answer save was silently denied ("Missing or insufficient permissions") for weeks; the exam still scored correctly at the end (scoring reads local React state, not the persisted `answers` array — see below), but answers didn't survive a closed tab mid-exam. Redeployed and confirmed fixed. Don't trust "the rule says X" claims about live behavior without confirming the deploy actually happened.
- **Any page calling `useSearchParams()` must wrap its content in a `<Suspense>` boundary** (App Router static-prerender requirement). Both `/auth/action` and `/auth/sign-in` now follow the same pattern: an inner `"use client"` component (extracted to its own file — `SignInForm.tsx` for sign-in) wrapped by a default-export server component inside `<Suspense fallback={...spinner...}>`. This pattern also satisfies the Next.js 15 rule that client components cannot be `async` (the server wrapper awaits `params` before passing resolved values to the client child). Originally fixed 2026-06-21 (commit `5bb5828`); extracted to separate file 2026-07-10 during the Next.js 15 upgrade.
- **Mobile-overlay panels in `Navbar.tsx` (mobile menu, user dropdown) must anchor to the viewport, not to their trigger button.** Below `lg`, the right side of the header is a packed icon row (avatar/streak/theme/language/hamburger), so a trigger near the left of that row has little room to its left — a panel sized with `absolute right-0` relative to the trigger's own small wrapper extends further left than the trigger and clips off the screen edge instead of centering under it. The user dropdown hit this 2026-06-21 (commit `49761fd`): `w-56` anchored to the avatar overflowed ~38px past the left edge, clipping the email/menu text (matches the same containing-block pattern already fixed for the mobile menu, immediately above). Fixed the same way: below `lg`, use `fixed left-4 right-4 top-16` (pinned to the viewport, same inset pattern as the mobile nav overlay) instead of `absolute`/`right-0` anchored to the trigger; `lg:` keeps the original trigger-anchored positioning, where desktop's wider viewport and lighter icon row make it safe.
- **`backdrop-filter` (Tailwind `backdrop-blur-*`) on an element makes it a containing block for its `position: fixed` descendants** — same rule as `transform`/`filter`, easy to forget since `backdrop-filter` reads as "just a visual effect." The sticky `<header>` in `Navbar.tsx` added `backdrop-blur-lg` to itself once scrolled (`scrolled` state, `scrollY > 10`); the mobile hamburger menu is a `fixed` child of that same header, so after scrolling it stopped sizing against the viewport and sized against the 64px-tall header instead, rendering as a thin cut-off strip instead of a full-screen overlay (only reproducible *after* scrolling, since unscrolled the header carries no blur — found 2026-06-21, commit `70c09a0`). Fixed by moving the blur to a separate absolutely-positioned background `div` that's a sibling of the `<nav>` and the mobile menu, never an ancestor, so `<header>` itself never carries `backdrop-filter`. Any future blur/frost effect on an element that also hosts (or whose descendants host) `position: fixed` content needs the same sibling-layer treatment.
- **A `position: sticky` + `z-index` ancestor (e.g. `<header>`) absorbs its `position: fixed` descendants into its own stacking context** — a descendant's own z-index only ranks it *within* that ancestor; compared to anything outside the ancestor, the whole subtree is ranked by the ancestor's z-index, not the descendant's. `Navbar.tsx`'s `<header>` is `sticky` + `z-50`; its mobile-menu overlay (a `fixed` child) is `z-40`. The global `BuyMeCoffeeButton` (a sibling of `<header>` at the root layout level, also `z-40`) looked on paper like a same-value tie a later-in-DOM element would win — but `<header>`'s own `z-50` dominates its entire subtree including the overlay, so the menu correctly paints over the button with zero code change needed (confirmed empirically in browser, 2026-06-21, after a code review raised this as a plausible-sounding but ultimately incorrect concern — see "Verified in browser" #13). Don't compare a sibling's z-index against a *nested* descendant's z-index without first checking which stacking context each one actually participates in.
- **Dashboard "Resume learning" CTA must pick the last-touched campus, not the first global module.** Module locking is off platform-wide (`ENFORCE_MODULE_LOCKING = false`), so every module across all 3 campuses is "unlocked" — a naive "first in-progress module, else first unlocked module" fallback (the original fix from item #4 below) lands on `unlockedModules[0]` whenever the user's only in-progress module reaches exactly 100%, which is just the first module of the global `CURRICULUM` array (`m1-typescript-foundations`, Automation) regardless of which campus the user actually used. Found 2026-06-21 via a real account (ISTQB progress only, with one fully-completed module) whose dashboard kept showing an untouched Automation module. Fixed (commit `c415cd8`) by adding `updatedAt` (ISO-8601) to each module's Firestore progress doc, written on every lesson-completion save (`updateUserProgress` in `firestore.ts`), and rewriting the picker (`pickResumeModule` in `dashboard/page.tsx`) to: pick the touched module (`completedLessonCount > 0`) with the most recent `updatedAt`; if it's already 100% complete, continue with the next incomplete module in the **same campus** (`getCampusForModule` + that campus's `moduleIds` order) before ever falling back to a different campus; only an account with zero progress anywhere falls back to the first module. Pre-existing progress docs lack `updatedAt` (no migration — they read as `null` and tie-break in `CURRICULUM` order until the user's next lesson completion populates a real timestamp), but the campus-scoping fallback already gets those accounts to the right campus immediately.
- **Campus page's "Progreso general" bar must read real progress, not a placeholder.** `CampusPageClient.tsx` (`/campus/[campusId]`) was shipped with that bar's text and `ProgressBar` value **hardcoded to `0`** — not a calculation that failed, just never wired up — so a campus already showing real progress on the dashboard's campus card (e.g. 11%/12%) always showed 0% once you opened it. Found 2026-06-22 via a real account (Test1: 11% QA Fundamentals, 12% ISTQB on the dashboard, both 0% on their campus pages). Fixed (commit `608fd66`) by adding `useAuth` + `useProgress` to the page and reusing the same `completedLessons / totalLessons` calculation the dashboard's per-campus card already used, filtered to that campus's `moduleIds` via `getModulesForCampus`. Any other "use client" page that shows a progress percentage should pull it from `useProgress`, never hardcode a placeholder value.
- **Campus page's module cards must show per-module completion state, not just the campus-wide bar.** Even after the overall-progress fix above, every module card in `CampusPageClient.tsx`'s grid rendered identically regardless of progress — a student who finished modules 1-2 and was starting module 3 had no visual cue which were already done (found 2026-06-23 via Jorge's own daily-use testing as a student, QA Fundamentals account with modules 1-2 complete). Fixed (commit `8c73cb8`) by calling `getModuleProgress(mod.id)` (already existed in `useProgress.ts`, already used by the dashboard's `ModuleCard.tsx`) per card and, when `status === "completed"`: switching the card to `Card`'s existing `"achievement"` variant (gold border/bg, same as badge/achievement cards elsewhere), appending a `✓` to the "Módulo N" tag, and coloring its per-module `ProgressBar` gold instead of green — exactly mirroring the dashboard's gold-on-completion convention (`ModuleCard.tsx:136`) rather than inventing a new visual language. No new colors/components introduced; module order and locking are untouched. A same-day follow-up (commit `a0d9c6c`) fixed a layout side-effect Jorge spotted comparing real cards side-by-side: title/description length varies per module, so the progress bar and lessons/minutes row landed at a different height on every card instead of lining up across the grid row. `Card` gained an optional `bodyClassName` prop (defaults to `""`, every other call site unaffected) so this page alone can make the card body `flex h-full flex-col`: the title+description block (`flex-1`) absorbs the leftover vertical space, and the progress-bar/stats block keeps a fixed `mt-4` gap and stays pinned to the bottom on every card regardless of content length. Confirmed in browser on both desktop and a 390px mobile-responsive viewport (see "Verified in browser" #16).
- **Leaderboard name/avatar sync uses `gamification/{uid}` as the single source, not a join.** The leaderboard queries only the `gamification` collection (`allow read: if isAuthenticated()` — any logged-in user can read any gamification doc). It cannot also read `users/{uid}` for fallback names because that collection is `allow read: if isAuthenticated() && isOwner(uid)` — owner-only, cross-user reads are denied by Firestore rules (a Permissions error surfaces in the UI if attempted). Instead, `displayName` and `photoURL` are kept current in the gamification doc via two write paths: (1) `AuthContext.syncProfile` does a fire-and-forget `updateGamificationDisplayName` on **every sign-in** so existing accounts whose gamification doc lacked a name are populated automatically the next time the user logs in; (2) `AuthContext.updateDisplayName` (Settings page) writes to all three stores — Firebase Auth, `users/{uid}`, and `gamification/{uid}` — so a name change is reflected in the leaderboard immediately. For the current user's leaderboard row the avatar reads `user.photoURL` from AuthContext (live from Firebase Auth, same source as the navbar) instead of the potentially-stale URL in the gamification doc; an `onError` fallback renders initials if any other entry's photo URL fails to load. `fetchLeaderboard(pageSize, lastDoc, rankOffset = 0)` takes a real `rankOffset` param for rank assignment (`rankOffset + i + 1`) — the current caller (`/leaderboard`, top 50, no pagination) doesn't need it, but a future "load more" page must pass the running count instead of leaving rank at `i + 1` for every page.
- **Email/password sign-up requires verification.** `signUpWithEmail` sends a Firebase verification email whose link opens our own `/auth/action` page (`actionCodeSettings.handleCodeInApp = true`); the `auth_token` cookie that `middleware.ts` checks is only set once `emailVerified` is `true`, so unverified accounts are bounced to `/auth/verify-email` (resend with 60s cooldown, auto re-check on tab focus) instead of the dashboard. `/auth/action` requires an explicit "Verificar mi cuenta" click before calling `applyActionCode` — it never fires automatically on page load, since email-client link prescanners (Outlook Safe Links, Gmail) silently visit and consume the one-time code otherwise. The validity of `mode`/`oobCode` is checked once in the initial `useState` (not a `useEffect`), so there's no render-then-effect gap that could flip the screen before a real click. If `applyActionCode` rejects a code whose account turns out to already be verified (e.g. a previous attempt consumed it), the page shows an "already verified" state pointing to sign-in instead of the generic invalid-link error. Google sign-in is exempt — Firebase marks Google accounts as pre-verified.

## Visual identity

- **Palette** (`tailwind.config.ts`, 50-900 scales): `brand.forest` (primary green), `brand.terra` (terracotta accent), `brand.gold` (achievements/ISTQB). Shades 300/400 are theme-aware CSS vars (`--brand-{forest,terra,gold}-{300,400}` in `globals.css`) so text stays legible in both themes.
- **Theme:** real light/dark toggle (`src/components/layout/ThemeToggle.tsx`, sun/moon icon in navbar), **defaults to light** (cream `#fbfaf5`, dark variant `#0e1814`). A no-flash inline script in `src/app/layout.tsx` reads `localStorage["theme"]` before paint. All theming flows through `--color-bg-*`/`--color-text-*`/`--color-border` CSS vars (`.light` = default/light, `:root` = dark) — `dark:` Tailwind variants are not used.
- **Typography:** `font-sans` Plus Jakarta Sans (body), `font-display` Chakra Petch (hero `<h1>`), `font-heading` Space Grotesk ("PlayQ" wordmark + campus card titles), `font-mono` JetBrains Mono (code).
- **Campus tiles** (`CampusCard.tsx` `CAMPUS_TILES`): `qaFundamentals` = forest (book), `istqb` = gold (certificate), `automation` = terra (automation icon).
- Landing: hero banner (`public/images/banner.jpg`) above the title; campus page has a collapsible exams panel (collapsed by default) so the modules grid is the focus.
- **Exams page (`/exams`):** single-open horizontal pill accordion, one pill per campus (name + exam count + chevron), `grid gap-3 sm:grid-cols-3` — the **same** grid classes as the stats cards row directly above it, so the two rows' columns line up; changing one without the other will misalign them. Clicking a pill expands that campus's exam list below and collapses any other open one; the page loads with all three collapsed. The per-exam card markup below the open pill (badges, "Comenzar" button, best-score line) is unchanged from before the accordion.
- **Curriculum certification section** (`/curriculum`, "Camino a la certificación"): exam cards are grouped into one block per campus (QA Fundamentals → ISTQB → Automation, `getSubCampuses()` order), each with its own forest/gold/terra heading and number-badge color (`CAMPUS_ACCENT` map in `page.tsx`, same convention as `CampusCard.tsx`'s tiles) — not one flat mixed-order grid. The "unlocks badge" hint under each card is derived at module load via `BADGE_BY_EXAM_ID` (every badge whose `criteria.type === "exam_passed"`, keyed by `examId`) — it shows the real badge name or renders nothing, never a guessed/hardcoded string.
- **Buy Me a Coffee button** (`src/components/layout/BuyMeCoffeeButton.tsx`, mounted globally in `src/app/[lng]/layout.tsx` as the last child after `<Footer />`): a `fixed bottom-6 right-6 z-40` brand-styled link (`bg-brand-terra-500`/`hover:bg-brand-terra-400`, the same accent as the navbar's "Crear cuenta" CTA — no new color introduced) to `https://buymeacoffee.com/jorgeborn3m`, opens in a new tab (`rel="noopener noreferrer"`). Self-contained: it owns its own `usePathname()`-based exclusion check rather than requiring callers to know about it, hidden on `/learn/[moduleId]/[lessonId]` (would overlap the lesson's sticky bottom nav bar), `/playground/dynamic` (would overlap the toast-locating exercise's own fixed corner), and `/exams/[examId]/take/[attemptId]` (avoids distraction during a timed exam). Any future global fixed-position element should follow this same self-exclusion pattern instead of hardcoding route awareness elsewhere.
- **Footer community links** (`Footer.tsx`): Instagram now points to the real PlayQAcademy account, `https://instagram.com/qatester.ai` (both the text-link list and the icon-only social row). GitHub and LinkedIn still point to the platform's generic landing page (`https://github.com` / `https://linkedin.com`), not a personal profile, since no dedicated account exists yet on those — swap in the real profile URL once one is created. `footer.learn.courses` points to `/` (home hub) — `/courses` was a dead route (404), there's no separate course-catalog page.
